```typescript
import { createClient as createServerSbClient } from '@/lib/supabase/server'
import { createAdminClient } from '@/lib/supabase/admin'

export default async function Debug() {
  const server = await createServerSbClient()
  const admin = createAdminClient()

  const { count: serverCount } = await server
    .from('products')
    .select('*', { count: 'exact', head: true })

  const { count: adminCount } = await admin
    .from('products')
    .select('*', { count: 'exact', head: true })

  return <pre>{JSON.stringify({ serverCount, adminCount }, null, 2)}</pre>
}
```

Why `_debug/` (underscore prefix):
- Next.js App Router treats `_` prefixed folders as private — they are NOT routable by default, but `app/_debug/page.tsx` IS routable because `page.tsx` overrides the private rule at its own level. Preferring `_debug` over `debug` keeps it clearly-marked as throwaway.

Expected render at `http://localhost:3000/_debug`:
- Both counts = 0 (empty tables, no user session). Value: `{"serverCount":0,"adminCount":0}`.

The page does NOT exercise `createClient` from `browser.ts` — that factory runs in Client Components only. Browser factory is verified indirectly by:
- TypeScript compilation (tsc imports it through import graph checks).
- Phase 2 will wire browser client into the auth modal (functional test happens there).

Do NOT:
- Make this page client-side — `createAdminClient` with `server-only` would break the build (that IS the guard test — see Task 4, not here).
- Query other tables — Phase 1 only has products + price_history.
- Leave the page in the codebase — Task 7 deletes it.
- Put the debug page in `src/` — `app/` is the routing root.
