## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env-var → SDK call site | `env.RESEND_TEST_RECIPIENT` (validated by Zod in Plan 09-01) flows directly to the `to:` field. No transformation, no parsing — Zod has already enforced email format. |
| caller (cron) → email module | `input.to` is the user-of-record's email, resolved by the cron orchestrator via `admin.auth.admin.getUserById`. Trust boundary is internal — both layers are server-only. |
| email module → Resend SDK | The four-field payload (`from`, `to`, `subject`, `html`) crosses to the Resend wire-level. Resend handles its own delivery / abuse mitigation; we do not double-validate. |
| log surface → Vercel function logs | `console.warn` writes to Vercel logs; the recipient address is operator-controlled (not user-input), so log-injection surface is low — but the structured-payload pattern (object, not template literal) is still applied per T-6-04. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-05 | Information Disclosure | module-load warn payload | accept | The warn payload contains the operator-set test-recipient email (`recipient: 'demo@example.com'`). This is not PII (operator's own demo address) and only appears in operator-controlled logs (Vercel dashboard, dev terminal). Acceptable for a portfolio-bar app. No production-user data leaked. |
| T-9-06 | Tampering | log-injection via env-var | mitigate | Warn uses structured object payload (`{ recipient: env.RESEND_TEST_RECIPIENT }`), NOT template-literal interpolation. Per T-6-04 / Phase 3 precedent. Even if an operator set a malicious env value, Zod `.email()` validation in Plan 09-01 rejects non-email strings at boot — the value never reaches the warn. |
| T-9-07 | Spoofing | unauthorized recipient routing | mitigate | The `??` expression has exactly two operands; both come from server-side trusted sources (env-var validated by Zod, or cron-orchestrator-resolved user email). No user-controlled input can influence the routing decision. |
| T-9-08 | Repudiation | who received the alert | accept | When override is set, the demo recipient receives all alerts; the user-of-record does not. This is the EMAIL-02 design intent; acceptable for portfolio-bar demo mode. The module-load warn provides the audit signal. Production mode (override unset) preserves v1.0 user-of-record routing exactly. |
| T-9-09 | Denial of Service | per-send warn flooding cron logs | mitigate | Module-load warn fires ONCE per process boot, NOT per `sendPriceDropAlert` call. Per-send warn is explicitly forbidden (CONTEXT Claude's Discretion). Cron logs stay clean even on volatile drop days. |
