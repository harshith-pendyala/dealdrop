    Final file contents — write exactly this:

    ```typescript
    import { NextResponse } from 'next/server'
    import type { NextRequest } from 'next/server'
    import { createServerClient } from '@supabase/ssr'

    export async function proxy(request: NextRequest) {
      let supabaseResponse = NextResponse.next({ request })

      const supabase = createServerClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
        process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
        {
          cookies: {
            getAll() {
              return request.cookies.getAll()
            },
            setAll(cookiesToSet) {
              cookiesToSet.forEach(({ name, value }) =>
                request.cookies.set(name, value)
              )
              supabaseResponse = NextResponse.next({ request })
              cookiesToSet.forEach(({ name, value, options }) =>
                supabaseResponse.cookies.set(name, value, options)
              )
            },
          },
        }
      )

      await supabase.auth.getClaims()

      return supabaseResponse
    }

    export const config = {
      matcher: [
        '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:png|jpg|jpeg|gif|webp|svg)$).*)',
      ],
    }
    ```

    Hard constraints — DO NOT violate:
    - Use `process.env.NEXT_PUBLIC_*` directly (NOT `import { env } from '@/lib/env'`). The Zod env module may not be Edge-runtime compatible — see Assumption A2 in 02-RESEARCH.md. This is the one file in the codebase that bypasses the env module.
    - Use `getClaims()` not `getUser()` (local JWKS validation, no Auth server round-trip per request). Verified in 02-RESEARCH.md.
    - `supabaseResponse` MUST be a `let` binding and MUST be rebuilt inside `setAll` before setting cookies on it. Returning a fresh `NextResponse.next()` at the end instead of `supabaseResponse` drops Set-Cookie headers — this is Pitfall 1 in 02-RESEARCH.md.

    ```typescript
    import { NextRequest, NextResponse } from 'next/server'
    import { createClient } from '@/lib/supabase/server'

    export async function GET(request: NextRequest) {
      const { searchParams, origin } = new URL(request.url)
      const code = searchParams.get('code')

      if (code) {
        const supabase = await createClient()
        const { error } = await supabase.auth.exchangeCodeForSession(code)
        if (!error) {
          return NextResponse.redirect(`${origin}/`)
        }
      }

    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /auth/callback | Attacker-controlled query string enters server trust boundary |
| Browser cookies → proxy.ts | Session cookies must be validated, not trusted blindly |
| Google OAuth → /auth/callback | OAuth `code` param is attacker-visible but single-use (PKCE-bound) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Spoofing | Session cookie in proxy.ts | mitigate | `supabase.auth.getClaims()` validates JWT signature against JWKS locally — spoofed cookies without a valid Supabase-signed JWT fail verification. |
| T-02-05 | Spoofing | OAuth code interception | mitigate | Supabase SDK uses PKCE — the `code` param alone is unusable without the code_verifier stored in the browser session. Attacker-redirect of `code` to a fake callback cannot complete the exchange. |
| T-02-06 | Elevation of privilege | /auth/callback open redirect | mitigate | Route Handler ignores any `next`/`redirect_to` query parameter (D-08 locked). Success redirect hardcoded to `${origin}/`, failure redirect to `${origin}/?auth_error=1`. The `origin` value is derived from `new URL(request.url)` — it's the host that received the request, not user-supplied. |
| T-02-07 | Tampering | Set-Cookie propagation in proxy | mitigate | `supabaseResponse` rebuild pattern in `setAll` ensures refreshed session cookies reach the browser on every eligible request. Pitfall 1 in 02-RESEARCH.md documents the regression mode (dropped Set-Cookie headers) we guard against. |
| T-02-08 | Information disclosure | Error details in callback redirect | accept | Failure path leaks only `?auth_error=1` — no stack trace, no Supabase error message, no OAuth provider error. Portfolio bar; richer error handling deferred to Phase 7 Polish per CONTEXT.md deferred items. |
| T-02-09 | Information disclosure | Service role key in proxy/callback | mitigate | Neither file imports `@/lib/supabase/admin` nor references `SUPABASE_SERVICE_ROLE_KEY`. Admin client is NOT used in Phase 2 (per CONTEXT.md `<code_context>`). |
</threat_model>
