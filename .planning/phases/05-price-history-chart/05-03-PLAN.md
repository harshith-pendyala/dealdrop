<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Component (DashboardShell) → Client Component (ProductCard → PriceChart) | Product data with nested price_history crosses the React Server Component → Client Component serialization boundary |
| Browser user → ProductCard `chartOpen` state toggle | Local React state; no persistence, no server round-trip per toggle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-03-01 | Information Disclosure | ProductCard → PriceChart prop drilling | accept | Only `product.price_history` (already RLS-scoped by Wave 1's nested select) and `product.currency` (public product attribute) are passed. No user_id, no session, no secrets cross the component boundary. Previous Phase 4 tests verified ProductCard does not leak user_id outside the auth context. |
| T-5-03-02 | Denial of Service | Toggle open → Recharts render | accept | Chart renders from pre-loaded props (D-01 eager load) — no server round-trip on toggle. Recharts render cost at portfolio scale (≤365 points) is ≤10ms. Risk accepted for portfolio-demo bar. |
| T-5-03-03 | Tampering | aria-hidden removed from chart slot | accept | Previous `aria-hidden="true"` on the placeholder div was appropriate (meaningless content). The chart IS meaningful content — removing aria-hidden exposes it to assistive tech correctly. No new surface; this is an a11y improvement, not a regression. |

**Security impact:** LOW — integration wave, no new trust boundaries, no new auth/data surfaces. Existing RLS + session cookie refresh (proxy.ts, Phase 2) continue to gate all data reaching PriceChart.
</threat_model>
