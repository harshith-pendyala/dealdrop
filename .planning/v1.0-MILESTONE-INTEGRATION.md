## DEP-06 Happy-Path Code Trace

The full sign-in → add product → cron → email → chart path was traced source-by-source.

### Step 1: AuthModal → /auth/callback (AUTH-01, AUTH-02, AUTH-05)

- `app/layout.tsx` mounts `<AuthModalProvider>` (wraps children), `<AuthToastListener>` (inside provider, Suspense-wrapped), and `<Toaster>` (outside provider but inside body).
- `AuthModal.tsx` calls `createClient().auth.signInWithOAuth({ redirectTo: ${window.location.origin}/auth/callback })` using the browser Supabase client.
- `app/auth/callback/route.ts` calls `await createClient()` (server factory), then `supabase.auth.exchangeCodeForSession(code)`, redirecting to `/` on success or `/?auth_error=1` on failure.
- `proxy.ts` matcher explicitly excludes `auth/callback` via negative-lookahead — no cookie race with `exchangeCodeForSession`.

**WIRED. No gaps.**

### Step 2: addProduct Server Action (TRACK-02, TRACK-03, TRACK-04, TRACK-05, TRACK-06)

- `ProductGrid.tsx` imports `addProduct` from `@/actions/products` and wires it via `useActionState` + `useOptimistic`.
- `actions/products.ts` (`'use server'` + `import 'server-only'` defense-in-depth):
  1. Calls `await createClient()` then `supabase.auth.getUser()` — re-checks auth on every action call.
  2. Calls `scrapeProduct(rawUrl)` from `@/lib/firecrawl/scrape-product`.
  3. Inserts into `products` (with RLS-enforced `user_id`) and into `price_history` (initial data point for TRACK-06/CHART-04).
  4. Best-effort rollback: deletes the `products` row if `price_history` insert fails.
  5. Calls `revalidatePath('/')`.
- `scrapeProduct` imports `env` from `@/lib/env.server` (server-only guarded), validates URL, calls Firecrawl v2, and returns discriminated result.

**WIRED. No gaps.**

### Step 3: ProductGrid useOptimistic (TRACK-08, DASH-01)

- `useOptimistic` in `ProductGrid.tsx` seeds from the `products` prop passed by `DashboardShell`.
- On URL submission, `addOptimistic(url)` fires _inside_ the `useActionState` async wrapper (React 19 transition boundary), inserting a `PendingItem` that renders as `<SkeletonCard>`.
- On action completion, `state` change triggers `dispatchToastForState` via `useEffect` with ref-deduplication.
- `AddProductDialog` re-uses the same `formAction`/`state`/`pending` from `ProductGrid` so dialog submits also trigger the skeleton.

**WIRED. No gaps.**

### Step 4: PriceChart render (CHART-01..CHART-06)

- `DashboardShell` (RSC) calls `getUserProducts()` which selects `*, price_history(price, currency, checked_at)` ordered by `checked_at ASC`.
- `Product` type from `@/lib/products/get-user-products` extends `Tables<'products'>` and adds `price_history: PricePoint[]`.
- `ProductCard` imports `PriceChart` and passes `product.price_history` + `product.currency`.
- `PriceChart.tsx` maps `dataKey="checked_at"` (X axis) and `dataKey="price"` (Line) — both match the `PricePoint` shape exactly.
- RLS ownership-chain (`price_history_select_own`) scopes history to the authenticated user's products through the server Supabase client.

**WIRED. No gaps.**

### Step 5: /api/cron/check-prices → runPriceCheck → sendPriceDropAlert (CRON-01..CRON-11, EMAIL-01..EMAIL-06)

- Route handler exports `maxDuration = 300`, `dynamic = 'force-dynamic'`, `runtime = 'nodejs'`.
- `GET` handler returns `{ status: 'ok' }` with no auth requirement (CRON-01).
- `POST` handler calls `verifyCronBearer(header, env.CRON_SECRET)` using constant-time `timingSafeEqual` before any scraping (CRON-02).
- `createAdminClient()` uses service-role key — bypasses RLS for reading all products (CRON-03).
- `runPriceCheck` (in `src/lib/cron/check-prices.ts`) uses `pLimit(3)` for bounded concurrency (CRON-04).
- Each product is processed by `processOneProduct` which calls `scrapeProduct` and handles: scrape failure (CRON-09), currency change guard, price unchanged (CRON-08 idempotency), price changed (CRON-07 — inserts `price_history` row + updates `products.current_price`).
- Email branch fires only when `scraped.current_price < product.current_price`. Calls `admin.auth.admin.getUserById(product.user_id)` to get recipient email (EMAIL-05). Calls `sendPriceDropAlert` with try/catch so network-level Resend throws don't abort the batch (EMAIL-06).
- `sendPriceDropAlert` in `src/lib/resend.ts`: imports `env` from `@/lib/env.server`, constructs HTML template with product image/name/old price/new price/percent drop/CTA link (EMAIL-02, EMAIL-03), calls `resend.emails.send`.

**WIRED. No gaps.**

## RLS Ownership Chain on price_history (DB-06)

**Code-level verification:**

1. Migration 0002 policy `price_history_select_own`: uses `product_id IN (SELECT id FROM public.products WHERE user_id = (select auth.uid()))` — ownership-chain correct.
2. User reads: `getUserProducts` uses the server Supabase client (scoped to `auth.uid()`). The PostgREST embedded resource `price_history(...)` inherits the ownership-chain RLS automatically.
3. Cron writes: `admin.from('price_history').insert(...)` uses service-role key — bypasses RLS entirely. No owned-user insert policy is required for cron.
4. User inserts (via `addProduct`): `supabase.from('price_history').insert(...)` uses the server client scoped to the authenticated user. The `price_history_insert_own` policy (0002) checks `product_id IN (SELECT id FROM products WHERE user_id = auth.uid())` — correct, prevents cross-user injection.
5. Cascade delete: `products.delete()` in `removeProduct` cascades to `price_history` via FK in 0001. RLS `products_delete_own` restricts this to own rows.

**CONFIRMED INTACT at code level. PITFALLS:339 operator confirmation is consistent with source.**

Path: pg_cron → `trigger_price_check_cron()` (Vault reads secret) → `net.http_post` → `POST /api/cron/check-prices` → `verifyCronBearer` → `createAdminClient()` → `runPriceCheck` → `processOneProduct` per product → `scrapeProduct` → price drop detected → `price_history` INSERT → `products` UPDATE → `getUserById` → `sendPriceDropAlert` → Resend email. **COMPLETE.** All connections verified. Chart update depends on page reload or `revalidatePath` (not real-time), which is per-design.

## Anti-Patterns and Gaps Found

### GAP-1: `proxy.ts` uses raw `process.env` instead of `@/lib/env`

**File:** `/Users/harshithpendyala/Documents/DealDrop/dealdrop/proxy.ts` lines 9–10  
**Pattern:**
```typescript
process.env.NEXT_PUBLIC_SUPABASE_URL!,
process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
```
**Expected:** `env.NEXT_PUBLIC_SUPABASE_URL` from `@/lib/env` (the established convention — "only `src/lib/env.ts` may touch `process.env`").  
**Impact:** Low severity. The non-null assertion (`!`) means a missing var would silently produce `undefined` at runtime instead of throwing the Zod validation message. In practice, these are `NEXT_PUBLIC_*` vars that the browser client also uses, so they will be present whenever the app builds. The vars are also validated in `env.ts` — any missing value would fail before reaching this path.  
**Affected:** FND-02 (partial — env validation convention is bypassed in proxy.ts). AUTH-07 (proxy session refresh depends on these being correct).  
**Severity:** Low. Works correctly at runtime; violates the single-reader convention.

### GAP-2: `app/favicon.ico` still present alongside `app/icon.tsx`

**File:** `/Users/harshithpendyala/Documents/DealDrop/dealdrop/app/favicon.ico`  
**Expected:** POL-06 requires favicon replaced with DealDrop asset. Plan 07-02 was supposed to delete `app/favicon.ico` and replace with `app/icon.tsx`.  
**Impact:** Next.js 16 will serve `icon.tsx` (dynamic "D" icon) as the preferred icon when both exist, so the branded icon is shown in practice. The `favicon.ico` is a stale artifact that creates ambiguity in the file tree and was meant to be deleted.  
**Affected:** POL-06.  
**Severity:** Low. Functional (branded icon renders); cosmetic cleanup omission.

### OBSERVATION: `price_history` insert in `addProduct` does not set `checked_at`

**File:** `/Users/harshithpendyala/Documents/DealDrop/dealdrop/src/actions/products.ts` line 53–57  
```typescript
await supabase.from('price_history').insert({
  product_id: product.id,
  price: result.data.current_price,
  currency: result.data.currency_code,
  // checked_at not set — relies on DB default now()
})
```
**Impact:** None. The `checked_at` column in migration 0001 has `default now()`. The DB default fires correctly. The cron handler also uses `nowIso` explicitly but the user-side insert intentionally omits it. CHART-04 ("at least one data point") is satisfied.  
**Severity:** Not a gap. Intentional reliance on column default.

### OBSERVATION: Toaster placement relative to AuthModalProvider

**File:** `app/layout.tsx` lines 35–42  
`<Toaster>` is placed _outside_ `<AuthModalProvider>` (sibling after the closing tag). `AuthToastListener` is _inside_ `<AuthModalProvider>` (uses `useAuthModal` context). Toast calls from `AuthToastListener` go directly to the Sonner singleton — they do not depend on Toaster being inside the provider tree. POL-01 is SATISFIED.  
**Severity:** Not a gap.
