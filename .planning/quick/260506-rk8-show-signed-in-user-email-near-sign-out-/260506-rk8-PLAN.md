  <behavior>
    Public API:
    ```tsx
    export const SCRAPE_STEPS: ReadonlyArray<{ label: string; atMs: number }> = [
      { label: 'Checking the link',          atMs:    0 },
      { label: 'Visiting the page',          atMs:  600 },
      { label: 'Reading the product details', atMs: 3500 },
      { label: 'Saving to your dashboard',   atMs: 7000 },
      { label: 'Finishing up...',            atMs: 9000 }, // hold-on-last
    ] as const

    export function ScrapeProgress({ pending }: { pending: boolean }): JSX.Element | null
    ```
    Behavior contract:
    - Returns `null` when `pending === false` AND no in-flight animation cleanup is needed (i.e. nothing to render).
    - On `pending` flipping false -> true: reset internal step index to 0 and start a timer chain (or interval) that advances `currentStepIndex` according to `SCRAPE_STEPS[i].atMs` deltas. Last step is the hold step.
    - On `pending` flipping true -> false: snap `progressPercent` to 100, render a brief "Done" frame for ~250ms (so user sees completion), then unmount (return null). Implementation may use a setTimeout scheduled in the same effect that fires when `pending` goes false.
    - Renders:
      - A horizontal progress bar (`<div role="progressbar" aria-valuenow={percent} aria-valuemin={0} aria-valuemax={100}>`) with an inner filled `<div>` whose width transitions to `${progressPercent}%` via Tailwind `transition-all duration-500 ease-out` (use `bg-primary` for the fill, `bg-muted` for the track, `h-1.5 rounded-full overflow-hidden`).
      - A vertical list of all steps (NOT just the current one). Each step has an icon slot:
        - completed steps: `<CheckCircle2 className="h-4 w-4 text-primary" />`
        - current step (pending): `<Loader2 className="h-4 w-4 animate-spin text-primary" />`
        - upcoming steps: `<Circle className="h-4 w-4 text-muted-foreground/40" />`
      - Step label text: `text-sm`; current step `text-foreground font-medium`, others `text-muted-foreground`.
    - Progress percent is computed as `(currentStepIndex / (SCRAPE_STEPS.length - 1)) * 100`. Step 0 -> 0%, step (length-1, the "Finishing up" hold) -> 100%. When pending flips false the bar snaps to 100% regardless of where the timer was.
    - Use `useEffect` with cleanup to clear all timeouts on unmount and on `pending` toggling (no leaked timers, no double-fire if user re-submits).
    - Use `setTimeout` chained per step (NOT a single `setInterval`) — simpler reasoning, matches the irregular `atMs` deltas.

    Required tests (use `vi.useFakeTimers()` + `act` from @testing-library/react):
    1. Renders nothing when `pending=false` initially (component returns null, container is empty).
    2. When `pending=true`, the first step label "Checking the link" is rendered immediately and its icon is the spinner (animate-spin class present).
    3. Advancing fake timers by 600ms advances the active step to "Visiting the page" (previous step now shows CheckCircle2 — assert by querying for the lucide CheckCircle2 svg by its class or by an aria-label/data-testid you add for the completed icon).
    4. Advancing past 9000ms holds on "Finishing up..." (the active step does not advance further; no error thrown).
    5. Re-rendering with `pending=false` after the bar started snaps progressbar `aria-valuenow` to 100 and, after the post-completion grace period (~250ms), the component unmounts (rerender returns nothing). Use `vi.advanceTimersByTime` to step through the grace period.
    6. The role="progressbar" element exists while pending=true and has `aria-valuemin=0` and `aria-valuemax=100`.

    To make icon assertions test-friendly, ALSO add a data-testid on each step's row (e.g. `data-testid={`scrape-step-${i}`}`) and a `data-state` attribute valued one of `"complete" | "active" | "upcoming"`. Tests can then assert state transitions without relying on lucide's internal SVG markup.
  </behavior>
